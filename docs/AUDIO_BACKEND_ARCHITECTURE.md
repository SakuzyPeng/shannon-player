# 音频后端架构约束

## 状态

- 决策状态：已接受，等待实现。
- 决策日期：2026-07-20。
- 适用范围：本地曲库扫描、音频解码、PCM 处理、设备输出、空间音频与链接导入。
- 当前版本仍使用前端种子数据和模拟播放状态；本文描述目标架构，不表示对应能力已经交付。
- 实现级设计（工程结构、线程模型、队列交接、测试策略与实施阶段）见
  [音频后端实现设计](AUDIO_BACKEND_IMPLEMENTATION_PLAN.md)。

## 决策摘要

Shannon Player 的实时播放链必须完全运行在应用进程内。播放、暂停、seek、解码、DSP、重采样、
音频设备输出和实时分析均不得依赖外部子进程。

子进程只允许出现在边界清晰、非实时的导入任务中：

1. 调用由用户自行安装和维护的 `yt-dlp`。
2. 下载结果确实无法直接导入时，调用 FFmpeg 完成必要的合并、重封装或转换。
3. 开发期诊断与离线验证工具，不进入正式播放路径。

不得把 FFmpeg PCM 管道、mpv/libmpv 或其他外部播放器作为隐藏的播放兜底。遇到不支持的格式时，
应选择可用的进程内后端，或返回明确的能力错误。

这里的“纯 Rust 解码”是主路径优先策略，不表示整条音频链拒绝原生 API。CoreAudio、WASAPI、
Media Foundation、PipeWire 等系统接口仍在应用进程内，属于允许的输出或平台能力后端。

## 分层

```text
React / Tauri 命令层
        |
        v
AudioService
        |
        +-- Source / Container
        |
        +-- DecoderBackend
        |     +-- SymphoniaDecoder
        |     +-- CodecSpecificDecoder
        |     +-- OptionalLibavDecoder
        |     `-- NativeSpatialDecoder
        |
        +-- PCM Pipeline
        |     +-- gapless 边界处理
        |     +-- ReplayGain / 音量 / DSP
        |     +-- 采样格式转换
        |     +-- 重采样
        |     `-- 声道映射
        |
        +-- 有界 PCM 环形缓冲
        |
        `-- OutputBackend
              +-- CpalOutput
              +-- PlatformExclusiveOutput
              `-- NativeSpatialOutput
```

容器识别、解码、PCM 处理和设备输出分别建模。文件扩展名、编码名称、声道布局和输出端点能力
不能合并成一个松散的“播放格式”字段。

## 进程边界

| 场景 | 是否允许子进程 | 约束 |
| --- | --- | --- |
| 播放、暂停、seek、切歌 | 否 | 全部由进程内播放器状态机驱动 |
| 实时解码、重采样、DSP、输出 | 否 | 不使用 FFmpeg/mpv 管道 |
| 曲库扫描、格式探测、标签与封面读取 | 否 | 使用 Rust 库或进程内原生库 |
| 波形、响度与媒体属性分析 | 否 | 后台线程执行，但仍在应用进程内 |
| 调用 `yt-dlp` 下载 | 是 | 用户自行安装；Shannon 只负责发现、调用、进度和错误呈现 |
| FFmpeg 合并、重封装或转换 | 条件允许 | 仅限无法直接导入的离线任务，并记录调用原因 |
| 不支持格式的播放兜底 | 否 | 不得静默启动 FFmpeg 或其他播放器 |
| 开发期诊断和概念验证 | 是 | 不得成为正式版本的运行时依赖 |

所有外部工具必须直接创建进程，不经过 shell 拼接命令。导入任务必须支持取消、清理未完成产物，
并把退出状态和标准错误转换为结构化错误；日志不得泄露 cookie、令牌或其他认证信息。

## 解码后端选择

后端按输入能力选择，不按“失败后不断尝试外部命令”的方式回退：

1. 带 Atmos/JOC 等对象元数据且平台提供已验证的空间解码能力时，选择原生空间后端。
2. 普通常见音频优先使用纯 Rust 解码器，当前首选候选为 Symphonia。
3. Symphonia 尚未成熟覆盖的编码，优先评估专用的进程内解码器，例如独立 Opus 解码器。
4. 只有格式覆盖确有必要且专用解码器不合适时，才评估基于 `ffmpeg-next` 的进程内 libav 后端。
5. 没有满足质量和能力要求的进程内后端时，返回“不支持的编码/容器”错误。

`OptionalLibavDecoder` 是可选扩展点，不是已确定依赖。是否引入必须同时评估格式收益、包体、
三平台构建、ABI 升级和与导入工具共享 FFmpeg 组件的可行性。

## 第一阶段格式范围

下表是实现与验证范围，不是当前版本的对外功能承诺：

| 输入 | 计划路径 | 说明 |
| --- | --- | --- |
| FLAC | Symphonia | 作为无损主力格式 |
| MP3 | Symphonia | 验证 encoder delay/padding 与无缝衔接 |
| WAV / AIFF / PCM | Symphonia | 保留采样格式、采样率和显式布局 |
| OGG Vorbis | Symphonia | 验证 chained stream 与 gapless |
| AAC-LC / M4A | Symphonia | 可解码；不能预先承诺完整 gapless |
| ALAC / M4A | Symphonia | 可解码；容器级 gapless 需单独验证 |
| Opus / WebM / Matroska | 专用进程内解码器 | 在纳入发布范围前以样本语料验证 |
| E-AC-3 JOC | Windows 原生空间后端 | 普通 PCM 解码结果不得标记为 Atmos |
| 固定 7.1.4 / 9.1.6 / 22.2 | 进程内解码器 + Windows 空间输出 | 布局来源和推断置信度必须保留 |
| 其他长尾或专有格式 | 待评估 | 不使用播放子进程兜底 |

Atmos/JOC 与固定多声道空间回放的证据、边界和平台限制分别见：

- [Windows Dolby Atmos/JOC 系统解码观测笔记](ATMOS_DECODING_NOTES.md)
- [Windows 固定多声道空间回放研究笔记](WINDOWS_SPATIAL_PLAYBACK_NOTES.md)

## 实时链路不变量

1. 音频接口统一以每声道时间帧建模，不能混用字节数、样本值数量和时间帧数量。
2. 解码线程与输出回调之间使用有界环形缓冲，内存不能随文件时长增长。
3. 实时输出回调不得进行文件 I/O、网络 I/O、阻塞锁等待、动态分配或重型日志记录。
4. 解码、重采样和较重的 DSP 在工作线程完成，输出回调只消费已经准备好的 PCM。
5. seek、切歌和重建后端时增加 generation；旧 generation 产生的 PCM 不得进入新播放位置。
6. 播放位置以已经提交到输出端的帧和设备时钟为准，不能以 JavaScript 定时器作为事实来源。
7. 普通切歌应预先打开下一条音轨并准备足够 PCM；支持 gapless 的格式不得因 UI 状态切换产生静音缝隙。
8. 设备切换、采样率变化和后端失效必须进入显式状态机，不能静默改用不同质量的输出路径。

## 输出后端

跨平台普通播放可从 CPAL 后端起步，负责设备枚举、配置协商和共享模式输出。以下能力不得假设
CPAL 已经统一提供，应由平台后端独立实现和探测：

- CoreAudio、WASAPI 等平台的独占或近似 bit-perfect 输出。
- 压缩码流直通。
- Windows Media Foundation Spatial Objects 解码。
- Windows `ISpatialAudioClient` 空间路由。
- 平台特有的设备热插拔、默认设备角色和错误恢复策略。

“解码 PCM 保持原始精度”与“最终设备 bit-perfect”是不同能力。只要发生重采样、DSP、共享混音、
声道映射或采样格式转换，就不能宣称 bit-perfect。

## 链接导入

链接导入属于独立的 `ImportService`，不直接参与播放器实时状态机：

```text
用户链接
   |
   v
用户安装的 yt-dlp 子进程
   |
   +-- 可直接播放的音频文件 ------------------+
   |                                         |
   `-- 必要时 FFmpeg 合并/重封装/转换 --------+--> 原子导入曲库
```

约束如下：

- Shannon 不分发、不自动安装 `yt-dlp`；界面负责发现版本、解释安装要求和呈现任务进度。
- 用户登录、cookie 和站点认证由用户自己的 `yt-dlp` 配置管理，Shannon 不接管账号凭据。
- 优先选择可以直接导入的原始音频流，避免为了统一扩展名而无条件调用 FFmpeg。
- FFmpeg/ffprobe 即使随应用提供，也只是导入工具资源，不得用于曲库扫描或实时播放。
- FFmpeg 只有在合并、重封装或用户明确请求转换时才可执行；每次执行都必须有可观测的原因。
- 下载完成后先在隔离临时目录验证，再以原子操作进入曲库；失败和取消不能留下半成品曲目。

## 明确不采用

- 不把 mpv/libmpv 作为主播放后端或正式分发依赖。
- 不以每首歌曲一个 FFmpeg 子进程的方式播放。
- 不用 `ffprobe` 子进程扫描本地曲库。
- 不把不支持格式静默转换后伪装为原文件直接播放。
- 不因为追求“全纯 Rust”而放弃已经验证的系统空间音频 API。
- 不把普通多声道 PCM、固定坐标槽位和带对象元数据的空间音频混为同一种能力。

## 验收条件

实现真实音频后端时至少验证：

1. 播放、seek、切歌、曲库扫描和媒体属性读取不会创建子进程。
2. 不支持格式返回带容器、编码和失败阶段的结构化错误。
3. 直接可导入的下载结果不会无条件调用 FFmpeg。
4. 必须调用外部工具的导入任务展示工具、阶段、进度、取消结果和可脱敏的错误信息。
5. 输出回调在持续播放、seek 和设备切换压力测试中没有阻塞分配，欠载指标可查询。
6. gapless、重采样、ReplayGain、设备切换和空间后端分别测试，不能只以“能够发声”作为通过条件。
7. 普通 PCM 路径不得展示 Atmos、动态对象或 bit-perfect 等未经证实的状态。

## 待定项

- Symphonia 的最终版本、feature 集与样本兼容性基线。
- Opus 进程内解码器的选型与多声道映射支持。
- 是否需要 `ffmpeg-next`，以及触发引入它的最低格式覆盖收益。
- CPAL 与平台专用输出后端的职责边界。
- 导入用 FFmpeg/ffprobe 的裁剪清单、共享库策略和三平台打包方式。
- 项目最低 Rust 版本；采用 Symphonia 0.6 时需从当前 `1.77.2` 提升到至少 `1.85`。

以上事项的当前建议汇总见[音频后端实现设计](AUDIO_BACKEND_IMPLEMENTATION_PLAN.md)的
「对架构约束待定项的建议」。
