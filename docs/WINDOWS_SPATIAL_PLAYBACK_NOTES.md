# Windows 固定多声道空间回放研究笔记

## 文档边界

本文整理独立概念验证中与 Shannon Player 音频后端有关的工程结论，关注已经解码为固定声道的
`7.1.4`、`9.1.6` 和 `22.2` 内容如何进入 Windows 系统空间音频渲染器。

它与 [Windows Dolby Atmos/JOC 系统解码观测笔记](ATMOS_DECODING_NOTES.md) 回答不同问题：

- Atmos/JOC 笔记关注系统能否从带对象元数据的码流中暴露超过 6 路 spatial/object buffer。
- 本文关注普通固定多声道文件的布局识别、流式解码和空间路由。
- 本文使用动态对象 API 提供固定坐标槽位，但输入不含对象坐标轨迹，也不能称为动态对象。

概念验证保持独立，没有把 C++ 源码、构建产物或第三方依赖引入本仓库。

## 验证目标

实验不依赖听感或实体多声道设备，主要验证：

1. 三种目标布局能否从 Opus-in-Matroska、WAV 和 LPCM CAF 流式得到正确声道数的 PCM。
2. 播放能否在 300 ms 预缓冲后开始，而不是等待整个文件解码。
3. seek 后旧位置的 PCM 是否会失效，并从新位置恢复提交。
4. 解码声道能否按显式布局进入 `ISpatialAudioClient`，不经过共享模式、下混或传统声道掩码猜测。
5. 在不监听最终声音的条件下，能否用渲染周期、对象缓冲提交和欠载等指标证明数据已经交给系统空间渲染器。

## 输入格式与布局识别

概念验证按文件签名分派容器，不只相信扩展名：

| 容器 | 支持的编码 | 布局来源 |
| --- | --- | --- |
| Matroska | Opus multistream，mapping family 255 | `OUTPUT_LAYOUT` |
| RIFF/RF64/BW64 WAV | PCM S16/S24/S32、F32/F64 | `bext` 中的约定字段 |
| CAF | 交错 LPCM，有符号整数或浮点，支持大小端 | `chan` layout tag |

目标布局标识包括：

| 布局 | 输入声道 | 已验证标识 |
| --- | ---: | --- |
| 7.1.4 | 12 | `7.1.4`、`4+7+0`、CAF Atmos 7.1.4 |
| 9.1.6 | 16 | `9.1.6`、CAF Atmos 9.1.6 |
| 22.2 | 24 | `22.2`、`9+10+3`、CAF CICP 13 |

布局处理遵循以下约束：

- Opus mapping family 255 只给出 multistream 映射，不定义扬声器语义，不能单独据此推导布局。
- WAV 传统 channel mask 无法完整表达三个目标布局，不能作为唯一依据。
- 显式标签存在但不受支持时必须拒绝，不能再按声道数把它误判成另一种布局。
- 只有完全没有显式标签时，概念验证才在限定范围内将 12、16、24 声道分别映射为三种目标布局。
- 生产实现应记录布局值和布局来源，避免把“容器明确声明”与“受限声道数推断”混为一种置信度。

## Windows 解码器兼容性

测试环境中的 Microsoft Opus Audio Decoder MFT 只接受单声道或双声道 Opus 输入。3 声道、
5.1、7.1 和三个目标样本均在 `SetInputType` 返回 `MF_E_INVALIDMEDIATYPE`。

这不是 mapping family 255 特有的问题，也不会因切换输出设备或开启系统空间音频而改变。因此概念验证采用：

- libwebm 增量解复用 Matroska。
- libopus 解码 Opus multistream。
- `dr_wav` 读取 RIFF/RF64/BW64 PCM。
- 结构化 chunk 解析器读取 CAF 的 `desc`、`chan` 和 `data`。
- 所有输入统一为 48 kHz、32 位浮点、交错 PCM，随后再交给布局路由层。

这个结果只描述被测 Windows 环境，不应推导为所有 Windows 版本上的永久限制。正式实现仍应先做能力探测，
再决定使用系统解码器还是进程内解码器。

## 流式读取模型

概念验证将容器、解码和渲染拆成三层：

```text
容器与解码器 -> 有界 PCM 环形缓冲 -> Windows 空间音频渲染器
```

统一音频流接口提供元数据、按帧读取、seek、缓冲帧数、读取单元数、EOF 和错误状态。关键参数为：

- 环形缓冲上限：96,000 帧，即 48 kHz 下 2 秒。
- 起播与 seek 后恢复阈值：14,400 帧，即 300 ms。
- PCM 后台读取块：4,096 帧。
- 24 声道时主 PCM 缓冲约 9.2 MB，内存不会随文件时长增长。

载入 Matroska 时只解析文件头和首个 cluster；WAV/CAF 只解析 chunk 表与音频描述。完整解码仅保留为
离线 `probe`，不属于 GUI 起播路径。

seek 行为按来源区分：

- Opus 清空环形缓冲、重置解码器，并按目标时间减去 seek pre-roll 后重新定位。
- 首次非零 Matroska seek 可以补建 cluster 索引，但只读取容器结构，不完整解码目标位置之前的 PCM。
- WAV/CAF 直接按 PCM 帧偏移定位。
- 每次 seek 增加 generation，旧 worker 产生的数据不能写回新位置的缓冲。

## 空间路由

三种布局的路由计划为：

| 布局 | 输入声道 | Windows 静态类型 | 固定坐标槽位 | 对象缓冲 |
| --- | ---: | ---: | ---: | ---: |
| 7.1.4 | 12 | 12 | 0 | 12 |
| 9.1.6 | 16 | 12 | 4 | 16 |
| 22.2 | 24 | 12 | 11 | 23 |

Windows 没有直接命名 9.1.6 和 22.2 全部扬声器位置的静态对象类型。缺少的位置通过动态对象 API
申请槽位，并在激活时设置一次固定坐标，播放期间不移动。

因此需要严格区分：

- **输入声道**：文件中固定布局的 PCM 声道。
- **对象缓冲**：每个系统渲染周期取得并写入的单声道缓冲。
- **固定坐标槽位**：借用动态对象接口承载、但位置保持不变的渲染槽位。
- **源对象**：码流中带元数据和轨迹的音频对象，本文输入不包含这一层。

22.2 有两个 LFE 输入，而 Windows 只提供一个静态 LFE 对象。概念验证按当前渲染块的峰值判断两路
活动状态，绝对值不超过 `1e-6` 的浮点残留按静音处理。只有一路含有效信号时以 unity gain 写入；
两路同时含有效信号时，才把两路各衰减约 3 dB 后合并到一个对象缓冲，避免单路内容被无故衰减。
这是明确的工程折中，不代表唯一正确的 22.2 渲染策略。

初始化端点时应同时检查：

- `ISpatialAudioClient` 能否激活。
- 所需静态对象掩码是否可用。
- `GetMaxDynamicObjectCount` 是否足以提供全部固定坐标槽位。

任一条件不满足时，应明确报告该端点不支持目标布局。概念验证没有静默切换到共享模式或下混路径。

## 无听感可观测指标

以下指标能够描述应用到系统渲染器之前的链路：

- **系统更新**：成功配对完成的 `BeginUpdatingAudioObjects` / `EndUpdatingAudioObjects` 周期数。
- **对象缓冲提交**：成功取得并写入的单声道对象缓冲总数。固定布局稳定播放时，应满足
  `对象缓冲提交 = 系统更新 × 对象缓冲数`。
- **已提交帧**：真正从解码 PCM 复制到对象缓冲的每声道时间帧数。
- **缓冲帧**：尚未被渲染回调消费的每声道时间帧数。
- **欠载**：播放状态下系统请求帧数超过环形缓冲可提供帧数的次数。
- **PCM 非零检查**：离线或流式测试中，整个读取块是否包含有限的非零样本，只用于排除读取失败或全零输出。

不应显示“X 个声道有信号”。曾使用的值是从最近一次 seek 开始、逐声道超过阈值的累计峰值数量：
它既不代表当前时刻，也不能可靠描述节目结构、布局占用或源对象数量。

这些指标可以证明多声道 PCM 已经进入系统空间音频流，但不能证明后续 HRTF、驱动或物理设备的最终声学结果。

## 验证结果

验证日期为 2026-07-16。正例基准包含 3 个 Opus MKA、3 个 WAV 变体和 3 个 CAF，覆盖三种布局：

- 每个正例均通过完整解码、流式读取与实际空间提交，共 27 个正向检查。
- 一个压缩型 CAF 和一个显式 HOA 布局 CAF 被正确拒绝，没有被声道数回退覆盖。
- 参考渲染器实际生成的一份 7.1.4 CAF 和一份 22.2 RF64 WAV 也通过流式读取与空间提交。
- 所有空间提交回归均完成至少 20 次系统更新，且欠载为 0。

2026-07-20 增加 LFE 条件合并回归：

- 合成 PCM 中仅第一路 LFE 有信号时，输出保持 unity gain。
- 仅第二路 LFE 有信号时，输出同样保持 unity gain。
- 两路同时有信号时，才应用约 -3 dB 等功率系数。
- 更新后的 22.2 Opus MKA、BW64 WAV 和 LPCM CAF 均再次通过空间提交测试，欠载为 0。

2026-07-21 使用另一份匿名完整 ADM BWF 渲染 108.50 秒的 7.1.4、22.2 Opus MKA 与
LPCM CAF。四个文件的完整解码、流式首读/中点 seek 和空间提交共 12 项验证全部通过，
空间测试欠载均为 0。22.2 CAF 的 LFE2 峰值为 -23.24 dBFS、RMS 为 -47.77 dBFS，
LFE3 的 5,208,000 帧全部为数字静音，再次覆盖单 LFE unity gain 路径。

CAF 保留精确的 5,208,000 帧；当前 Windows 播放器对两份 Opus MKA 的完整探测均报告
5,207,688 帧，少 312 帧，恰好等于 `OpusHead` pre-skip。MKA 容器时长仍为 108.50 秒，
首播、seek 和空间提交不受影响，但编码器末端补帧与容器 discard accounting 需要单独核对，
不能把本轮结果表述为 Opus 与 PCM 帧数完全相等。

流式性能观测如下：

| 来源 | 打开耗时 | 初始缓冲 | 中点 seek 等待 |
| --- | ---: | ---: | ---: |
| Opus MKA | 2 至 4 ms | 15,048 帧 | 19 至 25 ms |
| WAV/RF64/BW64 | 2 至 5 ms | 16,384 帧 | 0 至 2 ms |
| LPCM CAF | 2 至 3 ms | 16,384 帧 | 0 至 1 ms |

GUI 回归覆盖拖入、播放、暂停和约 70% 位置的 seek。seek 后重新预缓冲并恢复，播放位置、系统更新、
对象缓冲提交持续增长，欠载保持为 0。

独立程序还验证了静态链接部署：在没有相邻第三方 DLL 的空目录中，可以完成 Opus 流式读取、
WAV/CAF 空间提交和布局自检。这个结果证明单文件 PoC 可行，不约束 Shannon Player 最终采用的
Rust 音频依赖和打包方式。

## 对 Shannon Player 的实现约束

后续接入真实音频后端时，应保留这些边界：

1. 容器识别、编解码、布局解析和平台渲染分别建模，不能让扩展名或声道数承担全部语义。
2. 音频流接口以帧为单位，支持有界缓冲、异步 seek、generation 失效和可查询错误。
3. 布局使用强类型模型，并记录显式标签、推断来源和输入声道顺序。
4. Windows 空间渲染作为可探测能力，端点不满足时给出明确原因，不把下混结果标成沉浸声输出。
5. UI 区分输入声道、固定坐标槽位、对象缓冲和源对象；没有源元数据时不使用“动态对象”措辞。
6. 调试指标围绕缓冲、提交和欠载，不提供缺乏明确定义的逐声道活跃数量。
7. 完整文件分析与实时播放分开，播放器起播不依赖全量遍历。

## 当前未覆盖

- 非 48 kHz 输入与重采样。
- 7.1.4、9.1.6、22.2 之外的布局。
- APAC-in-CAF、非交错 CAF 和 WAV 中的非 PCM 编码。
- 对象坐标元数据解析、对象轨迹显示或自定义对象渲染。
- Windows 空间渲染器之后的 HRTF、驱动和物理设备验证。
- 22.2 双 LFE 的其他合并、分频或能量管理策略。
